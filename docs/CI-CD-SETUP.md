# CI/CD Setup for gcp-postgres-terraform + GitHub Actions + WIF

This guide explains how to set up a consuming repository to use gcp-postgres-terraform with GitHub Actions and Workload Identity Federation (WIF).

**Reference:** Also see [CLAUDE.md](./CLAUDE.md) for terraform module rules and variable reference.

---

## Prerequisites

Before following this guide, you must have:
- A GCP project with billing enabled
- WIF configured тАФ see the consuming repo's `docs/github-actions-wif-setup.md`
- GitHub repo with Actions enabled

---

## Required CI/CD Steps

When using this module, your GitHub Actions workflow must perform these steps in order.

### Step 1 тАФ Grant project-level IAM to the deploy SA

Run the WIF setup script once (see `docs/github-actions-wif-setup.md` in your consuming repo). This grants project-level roles including:
- `resourcemanager.projectIamAdmin` тАФ for terraform to read project state
- `serviceusage.serviceUsageAdmin` тАФ for listing enabled APIs
- `compute.viewer` тАФ for reading compute resources
- `storage.objectViewer` тАФ for reading bucket state
- `vpcaccess.admin` тАФ for reading VPC connector
- `monitoring.viewer` тАФ for reading dashboards/alerts
- `iam.workloadIdentityPoolAdmin` тАФ for managing WIF pools/providers

This is a one-time setup per SA. The SA name is `github-actions-deploy@<project>.iam.gserviceaccount.com`.

### Step 2 тАФ Pass the deploy SA to the postgres module

In your terraform, pass the SA email to the module:

```hcl
module "postgres" {
  source = "github.com/DarojaAI/gcp-postgres-terraform//terraform?ref=v1.10"

  project_id = var.project_id
  postgres_db_password = var.postgres_db_password
  instance_name = "my-pg"
  repo_prefix = var.repo_prefix
  environment = var.environment

  # Required for GitHub Actions + WIF
  github_actions_backup_reader_sa = "github-actions-deploy@${var.project_id}.iam.gserviceaccount.com"
}
```

The module creates `google_storage_bucket_iam_member.github_actions_backup_reader` (count = 1 when SA is passed) to grant the SA read access to the backup bucket. **However**, this requires a bootstrap step (see Step 3).

### Step 3 тАФ Grant bucket-level IAM (required for terraform plan)

**This is the critical bootstrap step.** The SA needs `storage.objectViewer` on the backup bucket so terraform plan can read bucket state. Project-level `storage.objectViewer` does not cover bucket-level `storage.buckets.getIamPolicy`.

The bucket is created by the postgres module on first apply. The SA cannot read bucket state until this step is run.

Add this step to your Terraform Apply GitHub Actions workflow **before running terraform plan**:

```yaml
- name: Grant bucket IAM for GitHub Actions SA
  run: |
    BUCKET_NAME="${SECRET_PREFIX}-postgres-backups"
    SA_EMAIL="github-actions-deploy@${PROJECT_ID}.iam.gserviceaccount.com"
    gcloud storage buckets add-iam-policy-binding "gs://${BUCKET_NAME}" \
      --member="serviceAccount:${SA_EMAIL}" \
      --role="roles/storage.objectViewer"
  env:
    PROJECT_ID: ${{ vars.PROJECT_ID }}
    SECRET_PREFIX: ${{ vars.SECRET_PREFIX }}
```

Set `PROJECT_ID` and `SECRET_PREFIX` as GitHub repository variables.

**Why this step exists:** Terraform plan must read the bucket's current IAM state to detect drift. The SA gets this permission on first apply (via the `google_storage_bucket_iam_member` resource the module creates), but the very first plan fails because the SA didn't have bucket-level read before apply ran. This step bridges the gap тАФ it runs as the SA (via WIF), so it works even if the SA has no bucket permission yet.

### Step 4 тАФ Terraform plan and apply

```yaml
- name: Terraform Plan
  run: |
    cd terraform
    terraform init -input=false
    terraform plan -out=tfplan

- name: Terraform Apply
  run: |
    cd terraform
    terraform apply -input=false tfplan
```

## Environment Variables and GitHub Vars

Configure these as GitHub repository variables (Settings тЖТ Variables тЖТ Actions):

| Variable | Example Value | Description |
|---|---|---|
| `PROJECT_ID` | `my-gcp-project` | GCP project ID |
| `SECRET_PREFIX` | `my-repo-prod` | Prefix matching `secret_prefix` in your terraform `.tfvars` |
| `ENV` | `prod` | Environment name |

---

## Bucket Name Pattern

The backup bucket name follows: `{repo_prefix}-{environment}-postgres-backups`

This is auto-generated by the module when `backup_bucket_name` is not explicitly set. The `SECRET_PREFIX` GitHub variable must match the `secret_prefix` in your terraform config тАФ they determine the bucket name.

## First Deployment Checklist

When deploying a new environment for the first time:

- [ ] WIF setup script run (project-level roles)
- [ ] `github_actions_backup_reader_sa` passed to postgres module
- [ ] GitHub repo variables set (`PROJECT_ID`, `SECRET_PREFIX`, `ENV`)
- [ ] Bucket IAM grant step in CI/CD workflow (Step 3)
- [ ] First apply succeeds тАФ creates bucket and all resources
- [ ] Future plan/apply cycles work without manual intervention

---

## Why These Steps Are Required

Without Step 1 (project-level roles): terraform plan fails on every GCP API read.

Without Step 3 (bucket-level grant): terraform plan fails specifically on `storage.buckets.getIamPolicy` when refreshing the backup bucket resource.

These are unavoidable bootstrap steps тАФ the SA must have bucket-level read permission before terraform plan can read the bucket, but the bucket only exists after the module's first apply creates it. The bucket-level grant bridges this gap.
